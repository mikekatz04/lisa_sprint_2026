# Contributing to the LISA Sprint 2026

This guide is the **sprint-wide** contribution reference for the
LISA Sprint 2026 work-tree. It covers the seven repos that sit side by
side under this directory:

```
lisa_sprint_2026/
├── GPUBackendTools/      # GBT — C++/CUDA base: cuda_complex, splines, FFT, CMake config
├── LISAanalysistools/    # LAT — central LISA library (response, WDM, TDIonTheFly, cutils)
├── GBGPU/                # GB sources, chunked-het, gbcomps, GB JAX/WDM
├── BBHx/                 # SOBBH sources, SOBBH JAX/WDM, MBH waveforms
├── FastEMRIWaveforms/    # FEW — EMRI templates (intentional cuda_complex fork)
├── Eryn/                 # MCMC / NUTS sampler
└── lisa-on-gpu/          # retired husk — 6.4 KB Python DeprecationWarning shim
```

Each repo also has its own `CONTRIBUTING.md` for repo-local conventions.
**This document covers the things that span repos** — the invariants,
the build wiring, the dev workflow, and the common pitfalls that
nothing inside a single repo can document on its own.

A schematic of the sprint architecture lives at
[sprint_architecture.png](sprint_architecture.png) (generated by
[sprint_structure_diagram.py](sprint_structure_diagram.py)). Open it
first if you have not seen the topology before.

> **Living document.** Sections are designed to grow. If you find a
> rule, a gotcha, or a recipe that is not written down, please add it —
> the goal is that nothing painful has to be learned twice.

## 2. Development environment

### 2.1 Conda env

Use `deving` (the canonical sprint dev env):

```bash
conda activate deving
```

Why `deving` and not base: the base miniconda env in this checkout has
a numpy 1.x / 2.x ABI mismatch with matplotlib. Anything that touches
numpy or matplotlib must run inside `deving`.

### 2.2 Build pin (mandatory)

**Every** `pip install` in the sprint must run with `PIP_CONSTRAINT`
pointed at the sprint constraints file:

```bash
export PIP_CONSTRAINT="$(pwd)/constraints/sprint.txt"
```

This pins both pybind11 and nanobind sprint-wide. The
process-global type registries of pybind11/nanobind are version-keyed —
if two wheels are built against different versions, classes registered
in one are opaque to the other, **silently**. Cross-wheel casts then
fail at runtime with confusing type errors.

Pin bumps are atomic across the sprint: change the version in
`constraints/sprint.txt`, rebuild every affected wheel in the same PR.
Per-package upgrades are forbidden.

### 2.3 macOS notes

- The CPU C++ build needs **Apple `clang` 17 from `/usr/bin/clang`**
  (or `brew install llvm@20`). Homebrew's default `clang` uses a
  `libc++` incompatible with the system runtime dylib and causes
  `symbol not found` at import.
- LAPACKE: `brew install lapack` and
  `export PKG_CONFIG_PATH="/opt/homebrew/opt/lapack/lib/pkgconfig:$PKG_CONFIG_PATH"`
  before installing. Or pass
  `--config-settings=cmake.define.GBT_LAPACKE_FETCH=ON` to fetch it.

### 2.4 GPU cluster notes (JAX)

On the GPU cluster, JAX needs:
- System `cuda12.6` module loaded
- `pip install "jax[cuda12_local]"`
- `nvidia-cudnn-cu12` wheel on `LD_LIBRARY_PATH` (the toolkit ships no cuDNN)

Mixing pip CUDA wheels with C++ backends built against system 12.6
causes `cuSolver handle create failed` errors at JAX import.

### 2.5 Installing the full chain

`install.sh` at the sprint root installs everything in editable mode in
the order GBT → LAT → GBGPU → BBHx → lisa-on-gpu → FEW. The order
matters: every wheel pulls headers/CMake config from the wheel
**before** it. When in doubt, run `install.sh` from scratch in a
fresh env rather than rebuilding piecemeal.

## 3. Sprint-wide invariants

These rules are enforced — by static asserts, grep gates, build-time
checks, or runtime tests. **Breaking one will cause hard-to-debug
failures across repo boundaries.** Read this section before your first
PR.

### 3.1 Backend implementation hierarchy

When implementing or modifying an algorithm that exists across multiple
backends (GPU C++ / CPU C++ / JAX):

1. **GPU C++ (CUDA) leads.** Canonical performance target and
   reference implementation.
2. **CPU C++ mirrors GPU C++ as closely as possible.** Same kernel
   structure, same algorithm. Use `#ifdef __CUDACC__` /
   `CUDA_SHARED` / `THREAD_START_X` / `BLOCK_INCR_X` to bridge.
   Must match lisatools reference output to machine precision
   (≤ 1e-15 mismatch) in direct modes.
3. **JAX may diverge internally** — design it to be JAX-efficient.
   Use `jax.lax.scan`, `jax.vmap`, **static-shape**
   `dynamic_slice` / `dynamic_update_slice`, functional accumulators.
   Do **not** mechanically translate the C++ kernel structure (shared
   memory, block tiling, register caches) — that produces slow JAX.
4. **JAX must match C++ inner-product outputs.** Likelihood quantities
   (`<d|h>`, `<h|h>`, the 5 swap_ll terms) match to reldiff ≲ 1e-12.
   Intermediate quantities (raw templates, per-chunk WDM coefs) may
   differ at FP precision due to summation order — validate at the
   inner-product level, not the intermediate level.

**Workflow for a new feature.** GPU C++ first → mirror to CPU C++ via
`#ifdef` → port to JAX with JAX-native idioms → validate each backend
against the others at the inner-product level **before merging**.

### 3.2 L1 — pybind11 / nanobind pin

`constraints/sprint.txt`. See §2.2.

### 3.3 L2 — single-registrant rule

Only LAT may call `nb::class_<>(m, "...")` for sprint-shared classes
(`Orbits`, `TDIConfig`, `WDMSettings`, `WDMDomain`, `FDDomain`, and
the LISA-response Wraps). Source-class wheels (GBGPU, BBHx) register
**only their own** classes (`GBTDIonTheFlyWrap`, `GBComputationGroupWrap`,
`SOBBHTDIonTheFlyWrap`, `SOBBHComputationGroupWrap`).

Enforcement is two-layer:

- The `LISATOOLS_IS_WRAPPER_OWNER` macro in
  `lisatools_header_abi.hpp` + per-TU `static_assert` fires at compile
  time if a non-owner repo defines it.
- `tools/check_single_registrant.sh` is a sprint-root grep gate that
  fails CI if `nb::class_` appears in a forbidden file. Extend the
  FORBIDDEN list when you add a new shared class.

If two wheels both register the same class, nanobind's global
registry sees a collision: the second registration is rejected or
silently shadows the first, and `isinstance` / downcasts break at
the Python boundary across wheels.

### 3.4 L3 — OrbitsView POD layout

`lisatools/cutils/orbits_view.hpp` defines the POD struct used to
share `Orbits` field offsets across wheels without re-`#include`-ing
the full `Detector.hpp` chain. The layout is asserted at every LAT
build via a `static_assert(sizeof + 15 offsetofs)` block in
`binding.cxx`. If you add or reorder fields in `Orbits`, update
`OrbitsView` in the same commit.

### 3.5 CPU/GPU class-name aliasing

Every C++ class compiled into **both** the CPU and the GPU wheels
must have a per-backend `#define` alias at the top of the header that
declares it, so the two builds emit **distinct C++ type names**:

```cpp
#if defined(__CUDA_COMPILATION__) || defined(__CUDACC__)
#define WDMSettings WDMSettingsGPU
#define WDMDomain   WDMDomainGPU
// ...
#else
#define WDMSettings WDMSettingsCPU
#define WDMDomain   WDMDomainCPU
// ...
#endif
```

Two layers need this:

- **Pybind11/nanobind wrapper classes** — anything passed to
  `nb::class_<...>(m, "...")`. Block lives in
  `binding_*.hpp` files.
- **Underlying C++ classes** the wrappers hold a pointer to — block
  lives at the top of each header that declares them
  (`Detector.hpp`, `TDIonTheFly.hh`, etc.).

Rules:

1. Every class — wrapper or underlying — that ends up in both `.so`s
   must appear in the relevant header's `#define` block, with both
   GPU and CPU branches.
2. Both branches must have the **same set of entries**. A missing
   branch silently produces a backend-asymmetric class name.
3. The `nb::class_<FooWrap>(m, "FooWrapGPU" / "FooWrapCPU")`
   registration in the `.cxx` binding source must be guarded by the
   same `#if defined(__CUDA_COMPILATION__) || defined(__CUDACC__)`
   toggle so the Python-visible name tracks the C++ alias.
4. Inheritance: if `class WDMDomain : public WDMSettings`, **both**
   names must be aliased. Otherwise the GPU build links
   `WDMDomainGPU` against the still-unaliased `WDMSettings` and
   collides across wheels.
5. File-static helper structs that never escape a single TU
   (e.g. a `.cu`-local `OrbitsSplineCache`) do **not** need aliasing.

**Rationale**: the CPU and GPU wheels load into the same Python
interpreter. Aliasing makes their exported symbols ABI-independent
and side-loadable — that is what lets `has_backend("cpu")` and
`has_backend("cuda12x")` both be true simultaneously.

### 3.6 Host → device upload of wrapper structs

Pybind11/nanobind wrapper classes (`OrbitsWrap`,
`TDIConfigWrap`, `WDMSettingsWrap`, `WDMDomainWrap`, `FDDomainWrap`,
…) store their underlying C++ struct via plain `new` on the **host
heap**. The pointer fields *inside* the struct may already point to
device memory, but the struct itself lives on the host.

A CUDA kernel parameter of type `Orbits *` therefore cannot be the
host pointer directly. Dereferencing it from device code reads
garbage and triggers an illegal memory access — typically with a
faulting address in the canonical PIE/heap range (`0x55555...`) and a
sanitizer message about being "N bytes after the nearest allocation"
with a TB-range delta. **That delta is not an off-by-one. It means
the device dereferenced a host address.**

Required pattern (see `lisa-on-gpu/src/.../LISAResponse.cu:419-433`
for the canonical implementation):

```cpp
#ifdef __CUDACC__
    Orbits *orbits_gpu = nullptr;
    gpuErrchk(cudaMalloc(&orbits_gpu, sizeof(Orbits)));
    gpuErrchk(cudaMemcpy(orbits_gpu, orbits, sizeof(Orbits),
                         cudaMemcpyHostToDevice));
    // ...repeat for every host-side wrapper struct accessed on device

    my_kernel<<<...>>>(orbits_gpu, /* ... */);
    cudaDeviceSynchronize();
    gpuErrchk(cudaGetLastError());

    gpuErrchk(cudaFree(orbits_gpu));
#else
    my_kernel(orbits, /* ... */);   // CPU branch keeps host pointers
#endif
```

The pointer fields *inside* the uploaded struct survive the shallow
copy — do not also try to upload those.

**Debugging heuristic.** IMA with `0x55555...` fault + TB-range
"nearest allocation" delta → first hypothesis is a missing wrapper
upload, not an indexing bug.

### 3.7 No backend strings as method kwargs

Backend choice is made at **object instantiation**, not as a kwarg
on individual methods:

- **Allowed**: `MyClass(force_backend="cpu")` /
  `MyClass(force_backend="cuda12x")` / `MyClass(force_backend="jax")`.
  Subclass `LISAToolsParallelModule` / `FastLISAResponseParallelModule`
  and dispatch via `self.backend` / `self.backend.xp` / `self.backend.name`.
- **Forbidden** in method signatures: `backend="jax"` / `use_cpp=True` /
  `use_jax=True`. Method names may carry a backend suffix
  (`get_ll_grad_jax`) when the implementation is intrinsically tied to
  that backend, but the choice belongs to the caller, not a runtime
  kwarg.

One instance = one backend, so its arrays / kernels / dispatchers
stay consistent. Mixing backends per call produces ambiguous
ownership of `xp` arrays, surprise host↔device copies, and brittle
dispatch logic.

## 4. Common recipes — "how do I…"

### 4.1 Add a new C++/CUDA class shared across backends

1. Decide where it lives. If it is generic LISA physics → LAT
   (`LISAanalysistools/src/lisatools/cutils/`). If it is sub-source
   physics (GB-specific, SOBBH-specific) → GBGPU or BBHx. If it is
   non-LISA-specific (interp, FFT, complex math) → GBT.
2. Add the class to its header. **Add it to the `#define` alias block
   at the top of the header**, with both GPU and CPU entries (§3.5).
3. If it's a wrapper (`nb::class_<>`), register it in LAT's
   `binding.cxx` only (L2 rule, §3.3). If it's GB-specific, register
   it in GBGPU's `binding_*.cxx`; SOBBH-specific in BBHx's.
4. If it has a `.cu` body, decide: header-only-inline, or `.hh + .cu`
   split. Phase 3L's pattern: small classes go header-inline; larger
   ones use `.hh + .cu` with LAT compiling the `.cu` into its own
   static archive (so the virtual-class typeinfo lives in LAT's `.so`
   — otherwise downstream imports `dlopen`-fail on missing `__ZTI*`
   symbols).
5. If a CUDA kernel takes this struct by pointer, audit every launch
   site for the host→device upload pattern (§3.6).
6. Extend `tools/check_single_registrant.sh`'s FORBIDDEN list if the
   class must not be registered elsewhere.

### 4.2 Modify a shared kernel

The chunked-het machinery in `lat_chunked_het_kernels.hh` and
`lat_wdm_fft.hh` is shared via `#include` between LAT, GBGPU's
`GBTDIonTheFly`, and BBHx's `SOBBHTDIonTheFly`. When you change a
kernel signature or behaviour:

1. Update the header in LAT.
2. Rebuild all downstreams that include it (`pip install -e
   LISAanalysistools/ && pip install -e GBGPU/ && pip install -e BBHx/`).
3. Update the JAX mirror at `lisatools/jax/wdm/heterodyne_kernels.py`
   + `computation_group.py` (and GBGPU/BBHx equivalents) to match
   **outputs**, not code (sprint rule §3.1).
4. Validate inner-product parity across CPU C++ / GPU C++ / JAX
   (§5.2).

### 4.3 Add a new Python module to LAT / GBGPU / BBHx

Put it in the relevant `src/<pkg>/` tree. Run any existing test files
in that package to confirm nothing breaks.

**Dev scripts (one-off, exploration, validation)** go in
`LISAanalysistools/scripts/<topic>/`, **not** the sprint root. Topics
already exist for `gb_chunked_het`, `gb_lookup`, `wdm`, `validation`,
`benchmark`, `emri`, `mbh`, `sobbh`, `diagnostics`. Add a new
sub-folder if your work does not fit an existing one.

The sprint root keeps a small set of intentionally-local helpers
(`_wait_then_fd.sh`, `sweep_knobs.sh`, `sobbh_intrinsic_Ladeeda.cpp`)
that stay where they are.

### 4.4 Add a new source class (e.g. a new GW source family)

Mirror the existing GB / SOBBH pattern:

1. C++ class derived from `LISATDIonTheFly` in the source wheel's
   `cutils/` (e.g. GBGPU's). Use the `.hh + .cu` split if non-trivial.
2. `*Wrap` + `*ComputationGroupWrap` registered in the source wheel's
   `binding_*.cxx`.
3. JAX path in the source wheel's `jax/sources/` mirroring the C++
   outputs.
4. Add the wrap to that wheel's backend dataclass
   (`GBGPUBackend` / `BBHxBackend`) so consumers reach it via
   `self.backend.MyNewSourceWrap`.
5. Frontend Python in LAT's response/TDIonTheFly hierarchy if the
   new source plugs into the existing chunked-het / FD / WDM flow.

### 4.5 Bump pybind11 or nanobind sprint-wide

1. Update the version in `constraints/sprint.txt`.
2. `export PIP_CONSTRAINT=...`.
3. `bash install.sh` from a fresh env. Every wheel rebuilds.
4. Run the cross-wheel smoke test (a simple
   `from lisatools.response import ...; from gbgpu import ...;
   from bbhx import ...`) to confirm the registry IDs agree.

## 5. Testing & validation

### 5.1 Per-repo tests

```bash
cd <repo>/
python -m pytest tests/
```

LAT's 56-test suite is the broadest cross-cutting check. After any
LAT change, run it before pushing.

### 5.2 Cross-backend inner-product parity

Required for any kernel or template change (§3.1.4). The standard
pattern:

```python
ac_cpu  = AnalysisContainer(..., force_backend="cpu")
ac_gpu  = AnalysisContainer(..., force_backend="cuda12x")
ac_jax  = AnalysisContainer(..., force_backend="jax")

ll_cpu = ac_cpu.template_inner_product(data, template, normalize=False)
ll_gpu = ac_gpu.template_inner_product(data, template, normalize=False)
ll_jax = ac_jax.template_inner_product(data, template, normalize=False)

assert abs(ll_cpu - ll_gpu) / abs(ll_cpu) < 1e-12
assert abs(ll_cpu - ll_jax) / abs(ll_cpu) < 1e-12
```

### 5.3 Narrowband mismatch (chunked-het / WDM)

When validating a chunked-heterodyne or other narrowband WDM template
against a lisatools reference:

- **`mm5`** — 5-layer mismatch over `[f0 - 3 layer_df, f0 + 2 layer_df]`.
  Primary chunked-het accuracy metric.
- **`mm2`** — 2-layer mismatch over `m_floor` and `m_floor+1`
  (`[(m_floor - 0.5) layer_df, (m_floor + 1.5) layer_df]`). Tighter
  check isolating the carrier.

Both are `1 - normalized overlap`:

```python
mm = 1 - <d|h> / sqrt(<d|d> <h|h>)
```

Canonical implementation:
`LISAanalysistools/scripts/gb_chunked_het/gb_chunked_prior_draws.py:283-340`.
Mirror this band definition for cross-source comparison.

**Current acceptance thresholds** (N_cp_sig=48, N_cp_orbit=32,
half-day wavelets, full angular prior):
- median mm5 ~ 1e-9, 90% < 8e-9, 99% < 3e-7
- low-frequency (`m_floor < 100`) sources occasionally show mm5 ~ 1e-7
  due to spectral-tail extension below `ind_min_f` — documented
  systematic, not a bug.

### 5.4 Single-registrant gate

```bash
bash tools/check_single_registrant.sh
```

Run this before pushing a C++ change that adds or moves a wrapper
registration.

## 6. Common pitfalls

| Symptom | Likely cause | Where to look |
|---|---|---|
| IMA with fault `0x55555...` and TB-range "nearest allocation" delta | Missing host→device wrapper upload | §3.6, audit your kernel launch site |
| `RuntimeError: type X already registered` at import | Two wheels registered the same class | §3.3, check the FORBIDDEN list |
| `isinstance(x, FooWrap)` is False even though it "should" be a `FooWrap` | Missing CPU/GPU alias entry on `FooWrap` or its base | §3.5, check the `#define` block has both branches and all parents |
| `dlopen` fails with `__ZTI<class>` not found at first downstream import | LAT did not compile the class's `.cu` into its archive (virtual-class typeinfo missing) | §4.1 step 4 |
| `cuSolver handle create failed` on JAX import on the cluster | Pip CUDA wheels mixing with system 12.6 backends | §2.4 |
| Apple `symbol not found` at C++ import on macOS | Built with Homebrew clang instead of Apple `/usr/bin/clang` | §2.3 |
| Inner-product matches CPU but not JAX | JAX path used `dynamic_slice` with **dynamic** shape (recompile thrash + numeric drift) — use static-shape slice + mask | §3.1 |
| `nb::cast<T>(x)` does not compile | nanobind: `.cast<T>()` is a **method** on the handle, not a free function | nanobind cheat-sheet (TBA) |

## 7. Authoring conventions

- Match the surrounding style; do not introduce new style preferences in
  the same PR as a behaviour change.
- Comments: explain **why**, not **what**. Identifiers and types
  already say what. Comments that paraphrase the code rot fast.
- Do not add fallbacks / shims / deprecation paths for code that has
  no current consumer. Backwards compat is for shipped APIs, not
  in-sprint internals.
- Editable installs only (`pip install -e`). Never `pip install .` —
  it breaks the inter-repo include chain that depends on each repo's
  in-place source tree.
- Commit messages: imperative mood, scope-prefixed
  (`LAT: ...`, `GBT: ...`, `Phase 3L.7: ...`).
